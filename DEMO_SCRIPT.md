# DEMO_SCRIPT.md — сценарий демо-видео

Задача: «Скажи мне, кто твой шлиф» (2026 Nornickel Hackathon, v2).

Формат: демонстрационный ролик ~5 минут (жёсткий верхний предел — 6:00).
Цель: за один просмотр показать, что решение (а) корректно ставит задачу,
(б) честно борется с утечками данных, (в) даёт измеримые метрики,
(г) работает как готовое локальное приложение, (д) воспроизводимо и разворачивается офлайн.

Язык озвучки: русский. Технические термины — на английском (SegFormer, IoU, F1),
интерфейс показываем в RU-локали (переключатель RU/EN в приложении есть).

> **Правило честности.** Все произносимые числа берём ТОЛЬКО из блока
> «Fact box» ниже. Это реально измеренные значения из `ChangeLog.md` и
> `docs/benchmarks/01_binary_sulfide_model_benchmark.md`. Ничего не округляем в
> большую сторону и не выдаём weak-supervision метрики за экспертную истину.

---

## Fact box (единственный источник произносимых цифр)

Сегментация сульфидов (SegFormer-B2, лучший чекпойнт, валидация на псевдо-метках):
- IoU `0.974`, F1 `0.987`, AUC `0.999`, HD95 ≈ `23.6 px`.
- Важная оговорка вслух: это **weak supervision** против псевдо-меток, а не
  экспертная геологическая истина.

Классификация сорта руды (image-level, ExtraTrees над агрегатами B2-фракций,
5-fold на deconflicted-сплите 345 изображений, 115 на класс):
- macro F1 `0.744`, accuracy `0.742`, macro AUC OVR `0.880`.

Тальк (SegFormer-B0, 5-fold, калиброванный):
- mean IoU `0.644`, mean F1 `0.782`.
- Оговорка вслух: разметка талька — примерные синие контуры, не сплошная маска;
  ~83% реального талька лежит вне исходных «мешков», поэтому маски дообучены на
  42 отревьюенных образцах.

Гигиена данных (аудит официальных папок):
- найдено `56` групп дубликатов по содержимому и `24` группы с конфликтом меток;
- финальная оценка идёт на deconflicted-сплите с dedupe + исключением конфликтов,
  сплит по образцу (group-level), чтобы метрики были честными.

Развёртывание:
- локально/офлайн, Docker (x86 VM и ARM64 gx10), холодный старт обычного
  изображения ≈ `18.6 s`, простой контейнера ≈ `301 MiB` RAM.

## Поблочный сценарий

### 0:00–0:30 — Постановка задачи
- **Экран:** заставка с названием задачи, затем крупный кадр оптического шлифа.
- **Действие:** плавный зум на вкрапления сульфидов и мутную тальковую зону.
- **Озвучка:** «Оценка аншлифов сегодня — ручная и субъективная. Нужно
  автоматически отнести образец к одному из сортов: обычные сростки,
  труднообогатимые тонкие сросты и оталькованные руды, где сорт определяется
  долей талька. Наше решение делает это по одному оптическому изображению.»

### 0:30–1:15 — Данные и честные метрики
- **Экран:** простой слайд с числами аудита (см. fact box) — 56 дубликатов,
  24 конфликта меток, group-level split.
- **Озвучка:** «Официальная разметка — на уровне изображения, а не пиксельные
  маски. Мы провели аудит и нашли 56 групп дубликатов и 24 группы с
  конфликтующими метками. Всё это убрано, а разбиение сделано по образцу — иначе
  метрики завышаются за счёт утечки. Метки талька — это примерные контуры, а не
  сплошные маски, поэтому мы относимся к ним как к слабой разметке.»

### 1:15–2:10 — Архитектура и метрики
- **Экран:** диаграмма пайплайна (из README):
  `panorama → binary sulfide segmentation → component features →
  ordinary/fine intergrowth → talc detection → ore-class rule → report`.
- Поверх — fact box с метриками.
- **Озвучка:** «Основа — SegFormer-B2 для бинарной сегментации сульфидов:
  IoU 0.97, F1 0.99 на валидации. Это метрики weak supervision против
  псевдо-меток, не экспертная истина — говорим прямо. По связным компонентам
  считаем признаки и классифицируем сорт: macro F1 0.74, AUC 0.88 на честном
  deconflicted-сплите. Тальк детектируем отдельной сегментацией с mean IoU 0.64,
  дообученной на 42 проверенных вручную масках. Итоговый сорт — детерминированное
  правило по долям сульфидов и талька.»

### 2:10–3:30 — Живое приложение: одиночный шлиф
- **Экран:** браузер, `apps/ore_pipeline_web.py`, страница `/workspace`.
- **Действия (по шагам):**
  1. Загрузить официальное изображение сорта.
  2. (опц.) Показать `Fix me` — пометку артефактов шлифовки до расчёта.
  3. Нажать `Start`, дождаться прогресса.
  4. Показать результат: текстовый вывод (сорт + доли + rationale-предложение),
     слои масок (сульфиды/несульфиды, обычные/тонкие сросты, тальк),
     непрозрачность и режим контуров, тепловую карту уверенности.
  5. Открыть иерархическую таблицу метрик (analyzed area, доли, число компонент;
     при калибровке масштаба — физические `mm²`).
  6. Нажать `Download ZIP` / открыть `report.pdf` — 5-страничный лабораторный отчёт.
- **Озвучка:** «Загружаем шлиф. При необходимости отмечаем артефакты подготовки,
  чтобы они не искажали доли. Запускаем — приложение выдаёт сорт, доли сульфидов
  и талька, маски по классам, карту уверенности и пояснение решения. Все прогоны
  неизменяемы, есть экспорт CSV и лабораторный PDF-отчёт.»

### 3:30–4:30 — Панорама (ключевой сценарий)
- **Экран:** та же UI, загрузка панорамного изображения (~27000×21000 px).
- **Действия:** запустить прогон → показать тайловую обработку с перекрытием →
  агрегированный сорт и долю талька → цветовую heatmap сорта по тайлам.
- **Озвучка:** «Панораму приложение обрабатывает тайлами с перекрытием и
  сшивает результат: агрегированный сорт, доля талька и тепловая карта сорта по
  участкам — так видно неоднородность образца.»
- **⚠️ Риск записи (обязательно к прочтению перед съёмкой):** живой панорамный
  прогон CPU-затратен; используйте образ с отложенной генерацией
  `preprocessed_full.png` (см. `docs/ui/v2/notes/2026-07-03-ore-pipeline-large-image-performance.md`)
  или запишите **заранее прогнанный** панорамный run и показывайте готовый
  результат из `/history`. Не снимайте многоминутное ожидание в реальном времени.

### 4:30–5:00 — Развёртывание и воспроизводимость
- **Экран:** терминал с командой запуска Docker + открытый PDF-отчёт.
- **Озвучка:** «Решение разворачивается локально и работает офлайн: Docker для
  x86 и ARM64, холодный старт обычного изображения около 18 секунд. Модели и
  правила зафиксированы, прогоны неизменяемы и воспроизводимы. Из ограничений —
  метки слабые, поэтому следующий шаг — экспертная валидация масок и калибровка
  правила сорта.»

## Команды для записи

```bash
# Основное приложение (надёжный heuristic-бэкенд для стабильного смока):
./run_main_app.sh --port 8080

# ML-бэкенд (когда доступно полное окружение и чекпойнт):
./run_main_app.sh --backend ml \
  --checkpoint models/binary_sulfide/segformer_b2_dataset_v0_zelda_20260703_overnight_safetensors/best.pt

# Калиброванное правило сорта для демо (после выбора артефакта калибровки):
#   добавить: --rule-config-json <.../ore_rule_calibration.json>

# Приложение ревью масок талька (при необходимости показать источник масок):
./run_talc_app.sh --port 8081

# Docker-путь (браузерный контейнер):
docker compose -f docker-compose.ore-pipeline-ui.yml up --build
```

Локально уже развёрнутые демо-инстансы (проверить перед съёмкой):
- x86 VM: `http://111.88.145.15:8080/workspace`
- gx10 ARM64 (LAN): `http://192.168.86.14:8210/workspace`
