# Plan 40: Real Panorama Compliance Card For Ore Pipeline UI

Date: 2026-07-04

## Scope

This plan is for the v2 OM-only ore pipeline UI and immutable run artifacts.
The goal is to turn existing large-image / tiled-inference telemetry into a
small judge-facing "panorama compliance" card for runs created from panorama or
panorama-scale images.

Primary targets:

- `apps/ore_pipeline_web.py`
- `apps/static/ore_pipeline_ui.html`
- run artifacts under `outputs/ore_pipeline_ui/runs/<run_id>/`
- PDF/ZIP exports generated by the ore pipeline UI
- focused UI tests and smoke checklist entries

This is not a plan to claim image-level grade accuracy on official panoramas.
The provided panoramas are unannotated and unclassified; the card is performance,
tiling, memory-safety, and visual-QA evidence.

## Current Inputs Already Available

The UI and pipeline already record most source data needed for the first card:

- `run.json.image.width/height`: analysis image size.
- `run.json.tiling`: source size, analysis size, tile size, stride, tile count,
  per-tile geometry, and `source_scaled_for_processing`.
- `run.json.preprocess.full_size_processing_deferred`: whether full-size
  preprocessing was intentionally skipped for a large source.
- `run.json.elapsed_seconds`: elapsed run time for current runs.
- `run.json.runtime` and `reports/runtime.json`: backend, checkpoint, model
  roles, device when available, thresholds, and completed timestamp.
- active-job `tile_progress`: processed/total ML tiles while a run is active.
- Status-page diagnostics: current RAM/disk/GPU snapshot, if the browser needs
  a runtime context outside the immutable run.

Known evidence points:

- The official dataset has 14 unlabelled panorama images; the largest is around
  `27025 x 21227` px.
- Local large-image smoke on `dataset/Панорамы/13.jpg` prepared
  `13330 x 9489 -> 1800 x 1282` in about `2.3 s` with
  `full_size_processing_deferred=true`.
- Pre-generated demo history includes panorama `13.jpg` runs, so the demo can
  show the card without waiting through a live run.
- The public Nornickel VM must not be used or redeployed after the 2026-07-04
  stop request; use local/gx10/pre-generated evidence only.

## Desired Jury Story

For a panorama run, the result panel should answer four questions immediately:

1. Did we process the real large source, or only a small sample?
2. How was it scaled/tiled for safe inference?
3. How long did the run take on the selected backend/device?
4. Is the evidence live, pre-generated, or missing timing/resource fields?

Example Russian copy:

```text
Панорама: источник 13330 x 9489 px -> анализ 2600 x 1851 px.
Тайлинг: 12 плиток, tile 1024 / stride 768, overlap 256 px.
Выполнение: ML B2, MPS, 13.5 c; full-size preprocessing deferred.
Статус: демонстрационная проверка производительности, не разметка ground truth.
```

## Artifact Contract

Each completed large-image run should write:

```text
reports/
  panorama_compliance.json
```

Recommended schema:

```json
{
  "schema_version": "ore-pipeline-panorama-compliance-v0.1",
  "applicability": "panorama_scale",
  "source": {
    "name": "13.jpg",
    "width": 13330,
    "height": 9489,
    "pixels": 126479370,
    "is_official_panorama_path": true
  },
  "analysis": {
    "width": 2600,
    "height": 1851,
    "scale_factor_x": 0.1950,
    "scale_factor_y": 0.1951,
    "full_size_processing_deferred": true,
    "source_scaled_for_processing": true
  },
  "tiling": {
    "enabled": true,
    "tile_size": 1024,
    "stride": 768,
    "overlap_px": 256,
    "tile_count": 12
  },
  "runtime": {
    "backend": "ml",
    "device": "mps",
    "binary_sulfide_model": "segformer_b2",
    "talc_source": "auto_candidate",
    "elapsed_seconds": 13.475,
    "started_at": "2026-07-03T21:19:06Z",
    "completed_at": "2026-07-03T21:19:20Z",
    "evidence_mode": "live_run"
  },
  "status": {
    "large_source_processed": true,
    "tile_manifest_present": true,
    "timing_present": true,
    "public_vm_used": false,
    "warnings": [
      "Official panoramas are unlabelled; this card proves processing/compliance, not class accuracy."
    ]
  }
}
```

Applicability rules:

- `panorama_scale`: source longest side exceeds the configured panorama
  threshold or `tiling.source_scaled_for_processing=true`.
- `normal_image`: card can stay hidden or collapse into "not a panorama run".
- `unknown`: show a warning when legacy runs lack source dimensions or timing.

## UI Contract

Add a compact card after the main result text/metrics and before lower-priority
tables, unless the active run is clearly a normal image. The card should be
visible on both loaded historical runs and freshly completed runs.

Fields to show:

- source size: `source_width x source_height px`;
- analysis size: `analysis_width x analysis_height px`;
- scale: percentage or `source -> analysis`;
- tiling: tile count, tile size, stride, overlap;
- runtime: backend, device, elapsed time;
- preprocessing: `full-size preprocessing deferred` yes/no;
- evidence mode: live run, loaded prepared run, legacy run missing timing;
- status/warnings: unlabelled panorama caveat and public-VM prohibition.

Behavior:

- If timing is missing on a legacy completed run, derive a best-effort duration
  from `runtime.started_at` and `runtime.completed_at` when both are present;
  otherwise show `timing unavailable` rather than `0 s`.
- If device is missing, show backend and checkpoint/source, then mark device
  unavailable.
- If the run is heuristic-only, still show source/tiling/preprocessing fields;
  do not imply ML 5-minute compliance.
- If the run was loaded from History, label it as pre-generated evidence when
  `completed_at` is in the past and no active job is attached.
- Do not show public VM hostnames as acceptable runtime evidence.

## Report And Bundle Contract

PDF report:

- Add a small "Panorama / large-image processing" block on the method/QC page
  when `reports/panorama_compliance.json` exists.
- Include the same caveat: panoramas are stress/performance evidence unless
  independently labelled.

ZIP/file browser:

- Include `reports/panorama_compliance.json` automatically through the existing
  run directory ZIP.
- File browser should list it like other JSON report artifacts.

Future optional screenshot:

- Save `reports/panorama_compliance.md` or a small PNG card only if the PDF or
  presentation needs a static extract. The JSON is the required artifact.

## Implementation Plan

### Phase 1: Builder And Legacy Compatibility

1. Add a stdlib-only helper in `apps/ore_pipeline_web.py`, for example
   `build_panorama_compliance_payload(run, run_dir)`.
2. Read fields from `run.json`, `runtime`, and `tiling`; do not re-open large
   image files unless source dimensions are missing.
3. Derive elapsed time from `elapsed_seconds` first, then
   `runtime.started_at/completed_at`.
4. Write `reports/panorama_compliance.json` during run finalization and lazily
   backfill it when loading a legacy run with enough metadata.
5. Add warnings instead of failing when optional runtime/device fields are absent.

### Phase 2: Result Panel Card

1. Extend the run payload returned by `GET /api/runs/{run_id}` with
   `panorama_compliance`.
2. Add localized RU/EN rendering in `apps/static/ore_pipeline_ui.html`.
3. Keep the card visually compact: one heading, two rows of key/value chips or a
   small table, and one caveat line.
4. Hide the card for normal images unless a user opens a "technical details"
   section; do show it for any run with source scaling or tiling enabled.
5. Make the card tolerate legacy runs where only `tiling` exists.

### Phase 3: PDF And Evidence Exports

1. Include the JSON artifact in `reports/` and therefore in `Download ZIP`.
2. Add a concise PDF section with source/analysis/tiling/runtime/caveat.
3. Ensure report generation for old runs does not fail if the compliance JSON is
   missing or partially populated.

### Phase 4: Optional Status Snapshot

Only after the base card works:

1. Capture a lightweight `resource_snapshot` at run start/end:
   - RAM total/available from the existing Status helpers;
   - GPU name/memory used if `nvidia-smi` is available;
   - disk free in the output root.
2. Store it in `panorama_compliance.json`.
3. Present it as supporting evidence, not a hard benchmark, because macOS/MPS
   and NVIDIA expose different counters.

## Tests

Focused unit/UI tests:

- builder returns `panorama_scale` when `source_scaled_for_processing=true`;
- builder hides/collapses as `normal_image` for small untiled images;
- elapsed time is derived from `runtime.started_at/completed_at` for legacy runs;
- missing device/timing produces warnings, not exceptions;
- `GET /api/runs/{run_id}` includes `panorama_compliance` for a panorama-scale
  run;
- generated ZIP/file list includes `reports/panorama_compliance.json`;
- PDF generation succeeds with and without the compliance payload;
- static UI exposes localized card labels and caveat strings.

Manual smoke:

1. Load prepared panorama run `13.jpg` from History.
2. Confirm the card shows source size, analysis size, tile count, backend/device,
   and pre-generated evidence wording.
3. Run a normal official image and confirm the card is hidden or collapsed.
4. Download ZIP and verify `reports/panorama_compliance.json` is present.
5. Open PDF and verify the panorama section appears only for large-image runs.

## Acceptance Criteria

- A panorama-scale run has a durable `reports/panorama_compliance.json`.
- The result panel displays a compact panorama card for panorama-scale runs.
- The card distinguishes source size from analysis size.
- The card reports tile size, stride, overlap, tile count, backend, device when
  available, and elapsed time when available.
- Legacy runs without timing show a warning instead of incorrect zero timing.
- PDF/ZIP exports include the card evidence.
- The UI explicitly states that official panoramas are unlabelled performance
  evidence, not ground-truth grade validation.
- The card never suggests using the public Nornickel VM after the stop request.

## Risks And Guardrails

- **Overclaim risk:** source-size/timing evidence can be read as accuracy proof.
  Always include the unlabelled-panorama caveat.
- **Legacy metadata risk:** older demo runs may have `runtime.started_at` and
  `completed_at` but no `elapsed_seconds`. Derive carefully and label provenance.
- **Backend comparability risk:** heuristic, MPS, CUDA, and resident batch timings
  are not directly comparable. Show backend/device next to every time.
- **Memory-counter risk:** macOS MPS lacks useful VRAM counters. Resource
  snapshots are optional and may be unavailable.
- **UI duplication risk:** if another technical-details widget lands, reuse its
  component/placement instead of adding a second bulky diagnostics block.

## First Practical Slice

1. Build and expose `panorama_compliance` from existing `tiling`, `runtime`, and
   `preprocess` metadata.
2. Render the compact result-panel card for panorama-scale runs.
3. Write `reports/panorama_compliance.json` and include it in existing ZIP/PDF
   evidence paths.
4. Add focused tests for payload generation, legacy timing fallback, and card
   visibility.
