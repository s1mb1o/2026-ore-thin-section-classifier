# Карта решения по официальному ТЗ «Скажи мне, кто твой шлиф»

Дата: 2026-07-02

Источник требований: `docs/official/Скажи мне кто твой шлиф.md` (официальная страница task-3, конвертирована 2026-07-02).

Назначение документа: для каждого шага и требования официального ТЗ зафиксировать, **как именно это решается** в нашем проекте. Плановая детализация и сроки — в `docs/plans/24_official-tz-gap-plan-claude.md` (воркстримы W1–W8) и `docs/plans/23_official-tz-gap-plan.md` (P0-0…P0-5).

Исполнительная площадка (по `docs/plans/25_standalone-ore-classifier-project.md`): ядро официального решения реализуется как отдельный узкий проект `submissions/ore_classifier/` (CLI `run_image.py` / `run_batch.py`). Строки `[готово]`, ссылающиеся на существующий портал (веб-интерфейс, коррекция маски, зум, метаданные), остаются в силе при опциональной интеграции движка в портал отдельной моделью; без неё их носителем становятся CLI, артефакты и README отдельного проекта — см. ревью в плане 25 (пункт 7, решение за пользователем).

Статусы:

- `[готово]` — уже реализовано и проверено в репозитории;
- `[доработка]` — инфраструктура есть, нужна адаптация под официальную задачу;
- `[новое]` — реализуется в финальном спринте (в скобках — воркстрим плана 24).

## 1. Проблематика — чем отвечаем на каждую боль

| Боль из ТЗ | Наш ответ |
| --- | --- |
| Субъективность оценки | Детерминированный пайплайн: одна и та же маска, доли и правило классификации на любых запусках; все пороги и параметры логируются, вывод воспроизводим между лабораториями. |
| Трудоёмкость (часы на образец) | Автоматическая сегментация панорамы целиком тайловым инференсом; целевое время — минуты на панораму (см. НФТ «Производительность»). |
| Масштабируемость (партии шлифов) | Пакетный режим без участия пользователя: очередь заданий, сводный `summary.csv` c классом руды по каждому образцу — готовая основа технологической карты. |
| Сложность количественной оценки «преобладания» | Вместо визуальной оценки — точные проценты: доля сульфидов, доли обычных/тонких срастаний внутри сульфидной площади, доля талька. |
| Вариативность данных (освещение, контраст, артефакты) | Предобработка (нормализация освещения, контраст, шумоподавление), маски исключения артефактов и отдельный сервис проверки устойчивости с вердиктом стабильности. |

## 2. Образ решения — как реализуется каждый шаг

### 2.1. Сегментация и классификация сульфидных включений `[новое]` (W3)

Двухэтапная интерпретируемая схема (выбрана исходя из состава официальных данных: у фото руд есть только метки класса на уровне изображения, попиксельных масок срастаний нет — см. `docs/notes/2026-07-02-domain-datasets-search.md`):

1. **Бинарная сегментация сульфидов**: светлые фазы на тёмной/серой матрице выделяются полноразрешающим тайловым инференсом (база — существующий `experiments/qc_pipeline/run_segmentation_inference.py` с тайлингом и сшивкой логитов; яркостная/модельная сегментация, предобучение на LumenStone S2/S1 как на прокси полированных шлифов).
2. **Классификация каждого включения** (connected component) по степени замещения нерудной фазой — по морфологическим признакам: доля тёмных включений внутри контура зерна (замещение), размер зерна, изрезанность границы, сплошность (solidity):
   - крупное, сплошное, с минимальным замещением → **обычное срастание**;
   - значительно замещённое нерудной фазой, ажурное/тонкое → **тонкое срастание**.
3. **Калибровка порогов** — на ~1050 официальных фото с метками классов (`Фото руд по сортам. ч1/ч2`: рядовые vs труднообогатимые/тонкие), с разбиением по образцам без утечек. Метрика — F1 классификации типа срастаний на уровне изображения.
4. Опциональный апгрейд (только если базовый путь готов): псевдоразметка фото откалиброванным пайплайном и дообучение плотной 4-классовой модели (SegFormer/ResUNet, YOLO11-seg как запасной вариант; обучение на gx10).

Схема сознательно интерпретируемая: каждый порог геолог может проверить глазами на конкретном зерне — это прямой ответ на требование «предметной точности».

### 2.2. Обнаружение и количественная оценка талька `[новое]` (W3)

- Обучающий сигнал — официальная папка `Области оталькования` (42 файла). Организаторы подтвердили формат разметки: **синие линии прямо на фотографиях** (чат капитанов, `docs/notes/2026-07-02-telegram-shlif-captains-chat.md`). Конвертер: выделение синей линии по цвету → замыкание контура → заливка → бинарная маска талька; пиксели самой линии исключаются из обучающего изображения (чтобы модель не училась на разметке); визуальный QA-оверлей для контроля каждой пары.
- На этих масках обучается/калибруется детектор талька как тёмной рассеянной фазы в нерудной матрице (текстурно-яркостные признаки или лёгкая сегментационная модель).
- Ошибка доли талька измеряется на отложенных размеченных парах; целевой порог — ±3% (см. НФТ «Точность»).

### 2.3. Экспертная логика классификации `[новое]` (W2)

Отдельный детерминированный модуль (`classify_ore_type`, развитие существующего `classify_microstructure.py`):

```text
если доля_талька > 10%            -> оталькованная руда
иначе если обычные >= тонкие      -> рядовая руда
иначе                             -> труднообогатимая руда
```

Зафиксированные соглашения (подтверждаем у экспертов на Q&A 2026-07-03):

- доля талька считается от анализируемой площади (без зон исключения/артефактов);
- «преобладание» — доли обычных/тонких внутри сульфидной площади;
- `> 10%` строго: ровно 10% — не оталькованная;
- равенство 50/50 → рядовая руда + предупреждение «на экспертную проверку».

Граничные случаи покрываются юнит-тестами; вывод не зависит от опционального LLM-нарратора.

### 2.4. Интерпретируемый результат `[доработка]` (W1, W5)

- **Цветовая маска**: зелёный = обычные срастания, красный = тонкие, синий = тальк — новый реестр классов (`ore_ontology`) подключается к существующему интерактивному просмотру результата (режимы Original/Overlay/Segmented, повключаемые классы) и к PNG-оверлеям.
- **Таблица метрик**: существующая таблица долей фаз (`create_sample_report.py`, знаменатель с учётом исключённых зон) перенастраивается на строки: общая доля сульфидов, доля обычных, доля тонких, доля талька.
- **Текстовый вывод**: детерминированная фраза в формате ТЗ — «Руда классифицирована как оталькованная: содержание талька — 14%, преобладание тонких срастаний — 62%» — в веб-результате, PDF и JSON/CSV.

## 3. Ключевые требования

| Требование | Статус | Как решается |
| --- | --- | --- |
| Предметная точность (геологическая логика, а не только метрики сегментации) | `[новое]` (W2, W3) | Финальный класс руды выдаёт явное экспертное правило, а не «чёрный ящик»; классификация срастаний — по геологически осмысленному признаку (степень замещения); все пороги видимы и проверяемы. |

| Интерпретируемость (визуальная проверка геологом) | `[готово]`+`[новое]` (W4) | Интерактивный просмотр маски по классам уже есть; добавляются карта уверенности (heatmap) и статистика по каждому зерну; редактор коррекций позволяет вручную проверить и исправить любой участок. |
| Практическая интеграция (от загрузки TIFF/PNG до экспорта отчёта) | `[готово]` | Веб-портал `run_upload_ui.py`: загрузка изображения → асинхронная обработка → результат с маской/таблицей → CSV/PDF; JSON API для встраивания в лабораторный пайплайн; CLI для скриптовых сценариев. |
| Возможность адаптации (transfer learning) | `[доработка]` | Обучающие рецепты SegFormer/ResUNet/YOLO уже в репозитории (локально и на gx10); коррекции эксперта экспортируются в обучающий манифест и CVAT/Label Studio; адаптер датасетов позволяет подключить данные другого микроскопа. |

## 4. Функциональные требования

### 4.1. Обработка изображений

| Требование | Статус | Как решается |

| Форматы TIFF, PNG, JPEG | `[готово]` | `input_loaders.py` + портал принимают tif/tiff/png/jpg/jpeg/bmp; для браузера TIFF конвертируется в PNG-превью тех же размеров. |
| Автоматическая предобработка: нормализация освещения, шумоподавление, коррекция контраста, масштабирование | `[новое]` (P1-2) | Отдельный модуль предобработки с логируемыми параметрами: выравнивание фона/освещения (retinex/CLAHE), мягкое шумоподавление, нормализация контраста; для панорам — тайловое масштабирование вместо глобального ресайза. ## 5. Нефункциональные требования

| Требование | Статус | Как решается |
| --- | --- | --- |
| Панорама до 10 000×10 000 px ≤ 5 минут (CPU/GPU) | `[новое]` (W4) | Тайловый инференс с батчингом тайлов на MPS/CUDA/CPU; ленточная агрегация вместо полноразмерного float32-буфера логитов (~2.8 ГБ на 10k×10k×7); бенчмарк на реальной панораме из официального пакета с записью времени/памяти в `docs/benchmarks/`. Целевое окружение — одна T4 (≈16 ГБ VRAM) на команду (подтверждено организаторами), поэтому бюджет памяти проектируется под T4. Отдельный риск: официальные панорамы (~96 МБ JPEG) могут быть сильно больше 10k×10k — декодирование ленточно, лимит Pillow поднимается осознанно. |
| Надёжность на «сложных» случаях (неравномерное освещение, царапины, загрязнения) | `[готово]`+`[доработка]` | Маски исключения (вход + правки эксперта) выводят артефакты из знаменателя долей; слой кандидатов дефектов (`detect_defect_candidates.py`) подсвечивает царапины/блики/загрязнения на ревью; предобработка выравнивает освещение; robustness-сервис даёт вердикт стабильности под возмущениями. |
| Ошибка доли талька ≤ ±3% | `[новое]` (W6) | Метрика `talc_fraction_abs_error` на отложенных парах из `Области оталькования` (разбиение по образцам); результат честно публикуется в `metrics.json/csv` и в презентации. |
| F1 классификации типа срастаний ≥ 90% | `[новое]` (W6) | F1 и AUC на уровне изображения по официальным меткам классов (рядовые vs тонкие/труднообогатимые), утечко-безопасный и по возможности равновзвешенный сплит; для сегментации — IoU и Hausdorff/HD95. Это целевые метрики продукционного решения, а не цель получить 100% на предоставленных данных; уточнение зафиксировано в `docs/notes/2026-07-03-official-metrics-and-panorama-split.md`. |
| Интерфейс, интуитивный для геологов; ручная коррекция маски (active learning) | `[готово]` | RU/EN портал с подсказками; модальный редактор коррекций: прямоугольник/полигон/клик SAM-2, назначение фазы, исключение региона с причиной, пересчёт с учётом исключений; правки сохраняются как обучающие патчи. |
| Локальное развёртывание для конфиденциальных данных | `[готово]` | Полностью локальный стек (без внешних вызовов), запуск одной командой; Dockerfile + smoke-проверка деплоя; модели из локального кеша. |
