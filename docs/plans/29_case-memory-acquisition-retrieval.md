# План 28: Накопительная память случаев и retrieval по подписи съёмки

Дата: 2026-07-03
Статус: proposal (к реализации)
Область: optical microscopy only, официальный ore-task. Расширяет P0-путь, не заменяет его.

## 0. TL;DR

Строим **детерминированную накопительную память проанализированных шлифов** (`case_ledger`) и слой **retrieval** поверх неё. Никакого RAG/LLM в ядре: «похожесть» и весь анализ считаются на объяснимых геологических признаках и автоматически выводимой «подписи съёмки».

Это закрывает пожелание ТЗ про метаданные, усиливает интерпретируемость («скажи мне кто твой шлиф» = сравни с уже изученными шлифами), напрямую бьёт по доменному сдвигу (разные микроскопы = разные домены) и даёт сильное демо «шлиф → его двойники + карта уверенности».

**Что это НЕ:** не RAG, не векторная БД, не LLM-решатель класса. LLM-нарратор возможен только опционально, отдельным заземлённым слоем сверху (см. §9), и никогда не считает доли и не решает класс.

## 1. Мотивация и привязка к ТЗ

- ТЗ, доп. пожелание: **«Поддержка метаданных: учёт информации о масштабе изображения, условиях съёмки, типе месторождения для контекстной адаптации модели.»** Сейчас в v2 этого нет вовсе.
- ТЗ, ключевое требование: **устойчивость к вариациям данных** (освещение, контраст, артефакты, разное оборудование). Память случаев + подпись съёмки дают механизм контекстной адаптации и честный OOD-флаг.
- Название кейса — «Скажи мне, кто твой шлиф». Retrieval «похожих шлифов» отвечает на это буквально и объяснимо.
- Синергия с критическим путём: батч-прогон официального набора всё равно нужен для метрики F1 (план gap-review). Тот же прогон **засевает** `case_ledger` ~1050 записями → холодный старт решён бесплатно.

## 2. Реальность данных (проверено 2026-07-03 на `data/`)

Это фундамент плана — проектируем под то, что реально есть, а не под воображаемые метаданные.

### 2.1. Состав

| Класс | ч1 | ч2 | всего |
| --- | ---: | ---: | ---: |
| рядовые (row) | 68 | 497 | 565 |
| тонкие/труднообогатимые (fine) | 68 | 417 | 485 |
| оталькованные (talc) | 42 | 87 | 129 |
| Области оталькования (blue-line) | 42 | — | 42 |
| Панорамы | | | 14 |

Дисбаланс сильный; ч1 — маленькая аккуратная партия, ч2 — основной объём. Панорамы до `27025×21227` (211 МБ).

### 2.2. Метаданные: чего НЕТ и что ЕСТЬ

**Нет:** рудник, тип месторождения, освещённость, масштаб (мкм/пиксель) — ни в EXIF, ни в сайдкарах, ни в именах. Это может появиться только как ручной ввод пользователя на будущих данных.

**Есть (и это автоматически выводимо, без ручного ввода):**

1. **Подпись съёмки (acquisition signature)** — различает ≥2–3 конфигурации:
   - `DSCN*` (632 файла): EXIF `NIKON E4500`, размер `2272×1704`.
   - часть нумерованных ч2 (`63.JPG`, `-4.jpg`): `4160×2768`, **EXIF отсутствует** → другой аппарат.
   - часть нумерованных (`2539589-1`): снова E4500 `2272×1704`.
   Признаки подписи: `(exif_make, exif_model, width, height, has_exif, цветовая статистика)`.
2. **Увеличение (magnification)** — в именах (`5x`, `10x`), но: только у ~67 файлов, у 632 `DSCN*` отсутствует, и есть ловушка — **кириллическая `х` против латинской `x`** (`10х` vs `10x`). Влияет на размер компонент → на пороги ordinary/fine.
3. **Стем/идентичность образца** — `2548036 5x 1` / `2548036 5x 2`, `2550375-1` / `2550375-3`: несколько снимков одного физического образца. Нужно для leak-free split и retrieval «тот же образец».
4. **Партия ч1/ч2** — разные распределения, holdout-кандидат.

### 2.3. Data-gotchas (заложить в реализацию сразу)

- Парсить увеличение с учётом **кириллической `х`**.
- Два размера изображений → нельзя предполагать один аппарат.
- EXIF `DateTime` обнулён (`0000:00:00`) — не использовать.
- EXIF `XResolution=300` — мусорный тег, не масштаб.
- ч1 и ч2 — разные партии; не смешивать в split.
- Возможны байт-идентичные файлы в разных классовых папках ч1/ч2 (конфликт меток) — контент-хеш их вскроет.

## 3. Область и не-цели

**В области:**
- сбор `case_ledger` из `data/` и из каждого прогона пайплайна;
- детерминированный feature-вектор образца (геологически осмысленный);
- подпись съёмки + кластеризация в домены съёмки;
- retrieval «похожие шлифы» (kNN);
- контекстная калибровка/нормализация по группе `(домен, увеличение)`;
- OOD/уверенность из расстояния до накопленной базы;
- drift-мониторинг по срезам;
- аудит датасета (дубликаты/конфликты меток) как побочный продукт.

**Не-цели (P0):**
- RAG / векторная БД / внешние сервисы;
- LLM в качестве решателя класса руды (правило остаётся детерминированным);
- обучаемый metric-learning эмбеддинг (P1, см. §10);
- ручной ввод рудника/освещённости как обязательный (только опциональное обогащение).

## 4. Модель данных

### 4.1. `case_ledger.parquet` / `.csv` — одна строка на изображение

| Поле | Тип | Источник | Назначение |
| --- | --- | --- | --- |
| `case_id` | str | хеш(path)+stem | первичный ключ |
| `path` | str | обход `data/` или прогон | ссылка на файл |
| `content_sha256` | str | хеш байтов | дедуп, конфликты меток |
| `phash` | str | perceptual hash | near-duplicate |
| `folder_label` | str | папка | `row`/`fine`/`talc`/`panorama`/`talc_annotation` |
| `batch` | str | путь | `ch1`/`ch2`/`panorama` |
| `sample_group` | str | стем | группа образца (leak-free split, «тот же образец») |
| `magnification` | float\|null | имя (Cyrillic-safe) | калибровка по группе |
| `acq_make`,`acq_model` | str\|null | EXIF | подпись съёмки |
| `width`,`height` | int | изображение | подпись съёмки |
| `has_exif` | bool | EXIF | подпись съёмки |
| `mean_r/g/b`,`std_r/g/b` | float | цветовая статистика | подпись/нормализация |
| `brightness_mean`,`illum_lowfreq_*` | float | яркость/лоу-фриквенси | доменный дескриптор |
| `acq_cluster_id` | int | кластеризация подписи | ключ группы адаптации |
| **feature-вектор образца** (см. §4.2) | float×N | пайплайн | retrieval, OOD |
| `ore_class`,`talc_fraction`,`sulfide_fraction`, `ordinary_fraction`,`fine_fraction` | | `OreSummary` | результат анализа |
| `needs_expert_review`,`warnings` | | `OreSummary` | флаги |
| `checkpoint_sha`,`pipeline_version`,`run_ts` | | прогон | провенанс/воспроизводимость |
| `user_mine`,`user_lighting`,`user_scale_um_px` | опц. | ручной ввод | обогащение (обычно пусто) |

Метаданные (folder/batch/mag/acq/stem/hash) заполняются **и без запуска модели** — обход `data/` даёт готовый аудит-леджер. Результатные поля дописываются после прогона пайплайна.

### 4.2. Feature-вектор образца (детерминированный, объяснимый)

Из `OreSummary` + `component_features.csv` (уже считаются в [component_analysis.py](../../src/ore_classifier/component_analysis.py)):

- `sulfide_fraction` (от анализируемой площади), `talc_fraction`;
- `ordinary_sulfide_fraction`, `fine_sulfide_fraction`;
- `component_count`, `ordinary/fine_component_count`;
- агрегаты по компонентам: median/p90 `area_px`, median `solidity`, median `compactness`, median `dark_inside_ratio`, `boundary_complexity`, плотность фрагментации (число мелких компонент / площадь).

Каждая ось — геологически читаема, поэтому «похожесть» объяснима словами, а не «косинус в латентном пространстве». Векторы нормируются (robust-scaling по накопленной статистике леджера, сохраняется в `ledger_scaler.json`).

## 5. Слои retrieval и анализа

### A. «Похожие шлифы» (kNN)
Для нового образца — k ближайших по feature-вектору (косинус/евклид на нормированных осях), с фильтром по `acq_cluster`/`magnification` при желании. Вывод:
> Ближе всего к 5 ранее изученным образцам (домен B, 10x): 4 из 5 — труднообогатимая. Тальк 11% ± 1.5%.

Артефакты: `similar_cases.json` (id, расстояние, класс, ключевые оси различия), `retrieval_panel.jpg` (текущий шлиф + 5 двойников с оверлеями). Переиспользует near-duplicate/uniqueness-примитивы из [curation.py](../../src/ore_classifier/curation.py).

### B. Контекстная калибровка/нормализация по группе
Группа = `(acq_cluster_id, magnification_bucket)`. По накопленной статистике группы:
- подбор порогов `ComponentRuleConfig` для этой группы (если калибровка их разделила);
- выравнивание цветовой статистики к эталону группы (FDA-style low-freq перенос) — безопасный ответ на доменный сдвиг (см. план про препроцессинг: нормализуем в домен, не ломая контракт модели).

Это retrieval **калибровочных приоров**, а не генерация. Приоры лежат в `group_calibration.json`.

### C. OOD / уверенность
Расстояние нового образца (подпись + feature-вектор) до ближайшего кластера/kNN базы. Если > перцентильного порога (напр. p95 внутрикластерных расстояний) → `out_of_distribution=true` → поднимает `needs_expert_review` в `OreSummary`. Честный ответ на «новый рудник/микроскоп».

### D. Drift-мониторинг
Срез леджера по `batch`/`acq_cluster`/`magnification`/`run_ts`: сравнение распределений (доли классов, яркость, sulfide_fraction). Отвечает на «шлифы домена B систематически чаще „тонкие“ — геология или артефакт микроскопа?». Артефакт: `ledger_drift_report.md`.

## 8. Пофазовый план (deadline-aware)

**Фаза 0 — аудит-леджер (S, самое ценное на единицу времени).**
`case_memory.py` + `build_case_ledger.py`: обход `data/`, метаданные + подпись + magnification + стем + хеши. Даёт: (1) фундамент, (2) готовый аудит датасета (дубликаты/конфликты меток) до метрик. Модель не нужна.

**Фаза 1 — feature-вектор + retrieval (M).**
Дописывание результатов из батч-прогона, нормировка, kNN «похожие шлифы», `similar_cases.json` + `retrieval_panel.jpg`. Демо-фича готова.

**Фаза 2 — OOD/уверенность (S).**
Расстояние до базы → `out_of_distribution` → `needs_expert_review`. Дёшево, честно, сильно для жюри.

**Фаза 3 — контекстная калибровка/нормализация по группе (M).**
Группировка `(acq_cluster, magnification)`, per-group приоры/нормализация. Делать только если Фаза 1 подтвердила расхождение групп (иначе не нужно).

**Фаза 4 — drift-репорт (S, опц.).**
Срезы по леджеру.

**Опц. — LLM-нарратор (см. §9)** только на локальной модели и только сверху.

## 9. LLM-нарратор — опционально, вне ядра

Если останется время и есть **локальная** модель (Ollama/llama.cpp — облачный API нарушает требование локального развёртывания для конфиденциальных данных):
- вход: детерминированные метрики ЭТОГО шлифа + `similar_cases.json` + заметки по протоколу;
- выход: заземлённый абзац-объяснение со ссылками на числа/двойников;
- запрет: не считает доли, не решает класс, не выдаёт числа, которых нет в summary;
- бонус-фича: NL-запрос к леджеру («оталькованные рудника X с тальком 10–12%») → text-to-filter → сводка.

## 10. P1 / стретч

- обучаемый metric-learning эмбеддинг (triplet) поверх компонентных признаков для более точного retrieval;
- эмбеддинги из энкодера сегментации как доп. ось похожести;
- STAPLE/Snorkel-подобная фузия при появлении ручной проверки;
- привязка масштаба (мкм/пиксель) → абсолютные площади, когда пользователь введёт scale.

## 11. Тесты

- парсинг magnification: `5x`, `10x`, `10х` (кириллица), `5x 2`, без magnification → `None`;
- стем/группировка: `2548036 5x 1` и `... 2` → одна `sample_group`; `2550375-1`/`-3`;
- подпись съёмки: E4500 2272×1704 vs 4160×2768 без EXIF → разные `acq_cluster` (на синтетике/фикстурах);
- хеши: два байт-идентичных файла в разных папках → один `content_sha256`, флаг конфликта меток;
- retrieval на синтетическом леджере: ближайший сосед из того же класса при близком векторе;
- OOD: далёкий вектор → `out_of_distribution=true`;
- апсерт по `case_id`: повторный прогон обновляет, не дублирует строку.

## 12. Риски и честные оговорки

- **Холодный старт:** до батч-прогона леджер содержит только метаданные (без результатов) — retrieval по результатам недоступен; засеваем прогоном официального набора.
- **Ручные метаданные почти всегда пусты** на выданном наборе — не строить на них ничего критичного; подпись съёмки выводима и заменяет их.
- **Feature-вектор наследует ошибки пайплайна** (некалиброванные ordinary/fine пороги) — retrieval полезен даже при этом (сравнение однородно), но «двойников» подавать как справочный сигнал, не как истину.
- **Связность истории:** держать это как детерминированный retrieval-слой; LLM (если будет) — подчинённый и заземлённый, отдельно от решения.
- **Провенанс:** каждая строка несёт `checkpoint_sha`/`pipeline_version`; при смене чекпоинта векторы несопоставимы — версионировать леджер.

## 13. Формулировки для презентации

> Система не просто классифицирует один шлиф — она сверяет его с накопленной базой ранее изученных образцов, подбирает калибровку под конфигурацию съёмки/увеличение и честно отмечает, когда образец не похож ни на что виденное.

> Метаданные съёмки мы извлекаем автоматически (аппарат, размер, увеличение, идентичность образца), а не требуем ручного ввода. Ручные поля (рудник, освещённость, масштаб) — опциональное обогащение, которое накапливается по мере работы.
